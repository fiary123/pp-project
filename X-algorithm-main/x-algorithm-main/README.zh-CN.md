# X 的 For You 信息流算法

本仓库包含驱动 X 平台 “For You” 信息流的核心推荐系统。它将网络内内容（来自你关注账号的内容）与网络外内容（通过基于机器学习的召回发现）结合起来，并使用基于 Grok 的 Transformer 模型对全部内容进行排序。

> **说明：** 该 Transformer 实现移植自 xAI 发布的 [Grok-1 开源版本](https://github.com/xai-org/grok-1)，并针对推荐系统场景做了适配。

## 概述

For You 信息流算法会从两个来源召回、排序并过滤帖子：

1. **网络内（Thunder）**：来自你关注账号的帖子
2. **网络外（Phoenix Retrieval）**：从全局语料中发现的帖子

这两个来源会被合并后统一排序，排序由 **Phoenix** 完成。Phoenix 是一个基于 Grok 的 Transformer 模型，会为每条帖子预测交互概率。最终分数由这些预测交互值按权重组合得到。

我们移除了系统中所有手工设计的特征，以及大多数启发式规则。真正承担核心建模工作的，是这个基于 Grok 的 Transformer。它通过理解你的交互历史（你点赞、回复、转发等行为）来判断哪些内容与你相关。

## 组件

### Home Mixer

**位置：** [`home-mixer/`](home-mixer/)

它是负责组装 For You 信息流的编排层。它基于 `CandidatePipeline` 框架，包含以下阶段：

| 阶段 | 说明 |
|-------|-------------|
| Query Hydrators | 获取用户上下文（交互历史、关注列表） |
| Sources | 从 Thunder 和 Phoenix 中召回候选内容 |
| Hydrators | 为候选内容补充更多数据 |
| Filters | 去除不符合条件的候选内容 |
| Scorers | 预测交互并计算最终分数 |
| Selector | 按分数排序并选出前 K 个 |
| Post-Selection Filters | 做最终可见性和去重检查 |
| Side Effects | 缓存请求信息供后续使用 |

该服务暴露了一个 gRPC 接口（`ScoredPostsService`），用于返回指定用户的排序后帖子结果。

---

### Thunder

**位置：** [`thunder/`](thunder/)

这是一个内存中的帖子存储与实时摄取流水线，用于追踪所有用户的最新帖子。它会：

- 消费来自 Kafka 的帖子创建/删除事件
- 为每个用户维护原始帖子、回复/转发帖子和视频帖子的存储
- 向请求用户提供其已关注账号的“网络内”帖子候选
- 自动清理超过保留期限的旧帖子

Thunder 能够以亚毫秒级延迟完成网络内内容查询，而无需访问外部数据库。

---

### Phoenix

**位置：** [`phoenix/`](phoenix/)

这是机器学习组件，主要承担两个功能：

#### 1. 召回（Two-Tower Model，双塔模型）

用于发现相关的网络外帖子：

- **User Tower**：将用户特征和交互历史编码为嵌入向量
- **Candidate Tower**：将所有帖子编码为嵌入向量
- **Similarity Search**：通过点积相似度检索 Top-K 帖子

#### 2. 排序（带候选隔离的 Transformer）

用于预测每个候选内容的交互概率：

- 输入包括用户上下文（交互历史）和候选帖子
- 使用特殊的注意力掩码，使候选内容之间不能互相关注
- 输出每种行为类型的概率（点赞、回复、转发、点击等）

更详细的架构说明见 [`phoenix/README.md`](phoenix/README.md)。

---

### Candidate Pipeline

**位置：** [`candidate-pipeline/`](candidate-pipeline/)

这是一个可复用的推荐流水线构建框架，定义了以下 trait：

| Trait | 用途 |
|-------|---------|
| `Source` | 从数据源获取候选内容 |
| `Hydrator` | 为候选内容补充额外特征 |
| `Filter` | 去除不应展示的候选内容 |
| `Scorer` | 计算排序分数 |
| `Selector` | 排序并选取前列候选 |
| `SideEffect` | 执行异步副作用操作（缓存、日志等） |

该框架会在可能的情况下并行执行 sources 和 hydrators，并支持可配置的错误处理与日志记录。

## 工作原理

### 流水线阶段

1. **查询补全（Query Hydration）**：获取用户近期交互历史和元数据（例如关注列表）

2. **候选召回（Candidate Sourcing）**：从以下来源获取候选内容：
   - **Thunder**：来自已关注账号的最新帖子（网络内）
   - **Phoenix Retrieval**：从全局语料中通过机器学习发现的帖子（网络外）

3. **候选补全（Candidate Hydration）**：为候选内容补充：
   - 帖子核心数据（文本、媒体等）
   - 作者信息（用户名、认证状态）
   - 视频时长（针对视频帖子）
   - 订阅状态

4. **打分前过滤（Pre-Scoring Filters）**：移除以下帖子：
   - 重复内容
   - 过旧内容
   - 用户自己发布的内容
   - 来自已屏蔽/静音账号的内容
   - 包含已静音关键词的内容
   - 用户已看过或最近已投放过的内容
   - 不符合订阅可见性要求的内容

5. **打分（Scoring）**：按顺序应用多个打分器：
   - **Phoenix Scorer**：使用 Phoenix Transformer 模型生成机器学习预测值
   - **Weighted Scorer**：将多个预测结果合成为最终相关性分数
   - **Author Diversity Scorer**：衰减重复作者的分数以提升多样性
   - **OON Scorer**：对网络外内容分数做额外调整

6. **选择（Selection）**：按分数排序并选出前 K 个候选内容

7. **选择后处理（Post-Selection Processing）**：对即将返回的帖子候选做最终校验

### 打分与排序

Phoenix 这个基于 Grok 的 Transformer 模型会对多种交互类型进行概率预测：

```text
Predictions:
├── P(favorite)
├── P(reply)
├── P(repost)
├── P(quote)
├── P(click)
├── P(profile_click)
├── P(video_view)
├── P(photo_expand)
├── P(share)
├── P(dwell)
├── P(follow_author)
├── P(not_interested)
├── P(block_author)
├── P(mute_author)
└── P(report)
```

**Weighted Scorer** 会将这些值合成为最终得分：

```text
Final Score = Σ (weight_i × P(action_i))
```

正向行为（如点赞、转发、分享）拥有正权重。负向行为（如屏蔽、静音、举报）拥有负权重，因此会压低用户大概率不喜欢的内容分数。

---

### 过滤

过滤发生在两个阶段：

**打分前过滤器：**

| Filter | 用途 |
|--------|---------|
| `DropDuplicatesFilter` | 去除重复帖子 ID |
| `CoreDataHydrationFilter` | 去除核心元数据补全失败的帖子 |
| `AgeFilter` | 去除超过阈值的旧帖子 |
| `SelfpostFilter` | 去除用户自己的帖子 |
| `RepostDeduplicationFilter` | 对相同内容的转发进行去重 |
| `IneligibleSubscriptionFilter` | 去除用户无权访问的付费内容 |
| `PreviouslySeenPostsFilter` | 去除用户已经看过的帖子 |
| `PreviouslyServedPostsFilter` | 去除当前会话中已经投放过的帖子 |
| `MutedKeywordFilter` | 去除包含用户静音关键词的帖子 |
| `AuthorSocialgraphFilter` | 去除来自已屏蔽/静音作者的帖子 |

**选择后过滤器：**

| Filter | 用途 |
|--------|---------|
| `VFFilter` | 去除已删除、垃圾、暴力、血腥等帖子 |
| `DedupConversationFilter` | 对同一会话线程的多个分支进行去重 |

## 关键设计决策

### 1. 不使用手工特征工程

系统完全依赖基于 Grok 的 Transformer，从用户交互序列中学习相关性，而不是手工设计内容相关特征。这显著降低了数据流水线和在线服务基础设施的复杂度。

### 2. 排序阶段的候选隔离

在 Transformer 推理过程中，候选内容之间不能互相关注，只能关注用户上下文。这样可以保证某条帖子的分数不依赖于同批次里的其他帖子，使分数更加稳定，也便于缓存。

### 3. 基于哈希的嵌入

召回与排序都使用多个哈希函数来进行嵌入查找。

### 4. 多行为预测

模型不是只预测一个单一的“相关性”分数，而是同时预测多种行为发生的概率。

### 5. 可组合的流水线架构

`candidate-pipeline` crate 为构建推荐流水线提供了灵活框架，具备以下特点：

- 将流水线执行与监控逻辑同业务逻辑解耦
- 并行执行独立阶段，并优雅处理错误
- 易于新增来源、补全器、过滤器和打分器
