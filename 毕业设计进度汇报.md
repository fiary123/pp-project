## 一、项目整体概述

本系统以"降低领养门槛、提升领养质量、构建持续陪伴关系"为核心目标，将大语言模型（DeepSeek）与传统规则引擎结合，构建了一套**多智能体协同**的宠物养护与领养平台。

系统采用前后端分离架构：
- **后端**：Python + FastAPI，业务层以 CrewAI 框架编排多个专业 AI 智能体
- **前端**：Vue 3 + TypeScript + Vite，共 10 个功能页面
- **数据库**：SQLite（关系型数据）+ ChromaDB（向量知识库）
- **AI 服务**：DeepSeek API（兼容 OpenAI 接口）+ Edge-TTS 语音合成

## 三、各智能体功能详解

### 3.1 CoordinatorAgent（顶层协调智能体）

**文件**：`src/agents/coordinator.py`
**核心职责**：整个对话系统的"大脑"，负责意图识别、会话状态管理、消息路由分发。

**工作流程**：
```
用户消息
   │
   ▼
读取会话状态（session_context）
   │
   ▼
_classify_intent()  ←── 调用 LLM 进行意图分类
   │
   ├── preference_collection  →  _mock_demand_agent()   收集居住/偏好信息
   ├── adoption_audit         →  领养资质评估服务
   ├── knowledge_qa           →  知识专家问答
   └── general_chat           →  通用对话兜底
   │
   ▼
返回结构化响应 { reply, ui_actions, stage }
   │
   ▼
持久化会话状态（支持多轮对话上下文）
```

**关键设计**：
- 会话状态（`session_context`）存储在内存字典中，以 `session_{user_id}` 为 key，支持多轮对话连续性
- 返回的 `ui_actions` 字段可驱动前端动态渲染（如弹出表单、跳转页面），实现后端与前端的"行为解耦"
- 意图分类使用 LLM 完成，支持自然语言的灵活表达，而非传统关键词匹配

### 3.2 PetExpert（首席宠物匹配专家）

**文件**：`src/agents/pet_expert.py` + `src/web/services/ai_service.py`
**核心职责**：基于用户自然语言需求，从数据库中检索并推荐最匹配的宠物。

**创新实现——两阶段追问式匹配（Innovation Point 1）**：

**第一阶段：需求显化（`get_match_followup_questions`）**
- 用户输入初始描述（如"想养一只适合公寓的猫"）
- LLM 分析描述中的**信息缺口**，自动生成 2-3 个针对性追问
- 追问带有结构化选项，降低用户填写门槛
- 示例追问：
  ```json
  [
    {"key": "activity", "question": "您希望宠物的活跃程度如何？", "options": ["安静陪伴型", "中等活跃", "活泼好动"]},
    {"key": "experience", "question": "您之前有养宠经验吗？", "options": ["无经验", "有1-3年经验", "资深养宠人"]}
  ]
  ```

**第二阶段：语义匹配（`get_smart_match`）**
- 将用户初始描述 + 追问答案合并，构建完整需求画像
- LLM 对数据库中每只宠物进行语义匹配打分，输出可解释的结构化推荐：
  ```json
  {
    "id": 3,
    "fit_score": 92,
    "reason": "辛巴性格温顺，与您的家庭环境高度匹配",
    "fit_advantages": ["不怕陌生人", "适合有小孩的家庭"],
    "potential_challenges": ["日运动量需求较高"],
    "mitigation": "建议每天安排30分钟散步"
  }
  ```
- **降级机制**：LLM 不可用时自动切换为关键词模糊匹配，保证接口可用性

**使用工具**：`pet_database_search`（SQLite 模糊查询）

### 3.3 AdoptionProfiler（领养资质评估多智能体群）

**文件**：`src/agents/adoption_profiler.py` + `src/web/services/assessment_service.py`
**核心职责**：这是系统最核心的创新模块，采用**五层架构**实现对领养申请的可解释性 AI 评估。

#### 五层架构详解

**L1 — 数据输入层**：收集申请人完整画像
- 申请人描述文字（自然语言）
- 量化条件：月预算、日陪伴时长、是否有养宠经验、住房类型
- 目标宠物信息：品种、名称
- 原有宠物情况（共处风险评估依据）

**L2 — 规则预筛层（`rule_engine_prescreen`）**：硬约束快速过滤

| 规则类型 | 触发条件 | 处理方式 |
|---|---|---|
| 硬约束 | 申请描述中含"房东不让养"、"禁止养宠" | `hard_block=True`，直接拒绝 |
| 经济风险 | 月预算低于品种基础成本60% | 加入 risk_flags，penalty_score += 40 |
| 时间风险 | 日陪伴时长 < 2小时 | 加入 risk_flags，penalty_score += 20 |
| 经验风险 | 新手申请高难度品种（边牧/哈士奇等） | 加入 risk_flags，penalty_score += 15 |
| 环境风险 | 高层公寓未封阳台 | 加入 risk_flags，penalty_score += 10 |
| 复核阈值 | penalty_score ≥ 30 | `need_manual_review=True` |

输出结构：
```python
{
    "hard_block": False,
    "risk_flags": ["经济风险：预算偏低", "时间风险：工作日陪伴不足"],
    "penalty_score": 60,
    "need_manual_review": True,
    "prescreen_summary": "规则层摘要文字..."
}
```

**L3 — LLM 语义推理层**：三个专业智能体并行分析

① **`get_encyclopedia_agent`（品种百科专家）**
- 输入：目标宠物品种名
- 职责：检索品种基准信息，建立"标准参照系"
- 输出结构化 JSON Baseline：
  ```json
  {
    "space_requirement": "中等（建议≥50㎡）",
    "exercise_level": "高（每日≥1小时户外运动）",
    "companionship_need": "高",
    "monthly_cost_range": "500-1000元",
    "beginner_friendly": false,
    "special_notes": "需要大量智力刺激，否则易产生破坏行为"
  }
  ```

② **`get_adoption_profiler_agent`（领养画像分析师）**
- 输入：申请人信息 + 品种基准报告
- 职责：核心评分智能体，进行五层次深度分析：
  1. 四维匹配评分（环境/经济/时间/经验）
  2. 申请动机真实性分析（语义一致性检测）
  3. 综合就绪度评分（`readiness_score` 0-100）
  4. 风险因子提取（维度 + 描述 + 严重程度）
  5. 个性化改善建议
- 输出决策：`Approved`（≥80分）/ `Manual_Review`（60-79分）/ `Rejected`（<60分）

③ **`get_cohabitation_risk_agent`（共处环境风险评估专家）**
- 输入：申请人现有宠物信息 + 目标宠物信息
- 职责：评估多宠共处的具体风险
- 四维分析：疾病传播风险、行为冲突风险、物理安全风险、隔离过渡建议
- 输出 Markdown 格式风险报告

**L4 — 多智能体协同层（CrewAI Crew 编排）**

三个智能体以**串联（Sequential）**方式组成 Crew：
```
品种百科专家 → 领养画像分析师 → 共处风险专家
     │               │               │
  品种基准       综合评分决策      风险防护建议
     └───────────────┴───────────────┘
                     │
               整合为完整报告
```

**L5 — 输出与审计层（`assessment_service.py`）**

最终评分融合公式：
```
readiness_score = 规则层基础分（60分起） × 0.4 + LLM综合分 × 0.6
```

决策阈值：
- ≥ 80分 → `Approved`（系统自动通过）
- 60-79分 → `Manual_Review`（提交人工审核队列）
- < 60分 → `Rejected`（明确拒绝并给出改进建议）

所有评估过程写入 `agent_trace_logs` 表，管理员可通过 `/api/admin/assessment/report/{trace_id}` 查看完整的 AI 决策链路，解决"黑盒"问题。

### 3.4 NutritionExpert（宠物营养与喂养专家）

**文件**：`src/agents/nutrition_expert.py` + `src/agents/nutrition_planner.py`
**核心职责**：基于宠物个体参数生成科学、可量化、可执行的喂养方案，并支持基于反馈的迭代优化。

**科学计算流程（`build_nutrition_plan`）**：

```
输入参数：物种、月龄、体重(kg)、绝育状态、活跃度、目标(维持/减重/增重)、食物热量密度
       ↓
Step 1: 基础能量需求 RER = 70 × weight_kg^0.75  （千卡/天）
       ↓
Step 2: 日能量消耗 DER = RER × 活跃度系数 × 目标系数 × 年龄系数 × 绝育系数(0.9)
       其中：活跃度系数（低0.8 / 中1.0 / 高1.4）
             目标系数（减重0.8 / 维持1.0 / 增重1.2）
             年龄系数（幼崽≤6月: 1.5 / 青年≤12月: 1.3 / 成年: 1.0）
       ↓
Step 3: 每日喂食量(g) = DER / 食物热量密度 × 100
       ↓
Step 4: 喂食频次（月龄递减原则）
       ≤6月: 4次/天 | ≤12月: 3次/天 | >12月: 2-3次/天
       ↓
Step 5: 饮水建议 = 50-60 ml/kg/天
       ↓
Step 6: 生成禁忌食物列表（物种特化）
       通用禁忌：巧克力、洋葱、葡萄、木糖醇、酒精、高盐高油食物
       猫咪专属：避免长期以狗粮替代猫粮（缺乏牛磺酸）
       犬类专属：熟骨头（易碎裂刺穿肠道）、夏威夷果
       ↓
Step 7: 生成7日换粮过渡计划
       第1-2天: 25%新粮 + 75%旧粮
       第3-4天: 50%新粮 + 50%旧粮
       第5-6天: 75%新粮 + 25%旧粮
       第7天:   100%新粮
       ↓
输出：结构化方案（JSON）+ Markdown 格式喂养报告
```

**闭环优化机制（创新点3——数据飞轮）**：
```
初始方案生成
     ↓
用户执行 1-2 周
     ↓
提交反馈（体重变化/食欲/排便/活动量）
     ↓
replan_nutrition() 调用 LLM 分析反馈
     ↓
生成优化后的新方案（调整热量 ±10-20%）
     ↓
再次执行 → 再次反馈（持续闭环）
```

**使用工具**：`calc_pet_daily_energy`（封装营养计算），`pet_food_forbidden_list`（禁忌食物查询）

### 3.5 MedicalExpert + NavigatorAgent（医疗分诊专家 + 导航员）

**文件**：`src/agents/medical_expert.py` + `src/agents/navigator.py`
**核心职责**：症状初步分诊 + 就近医院导航，构成完整的紧急响应链路。

**MedicalExpert 工作流**：
```
用户输入症状描述（文字）
       ↓
pet_health_knowledge_search() ← ChromaDB RAG 检索（优先）
       ↓ （若向量库未返回结果）
_fallback_health_search()      ← 本地20+症状字典（兜底）
       ↓
LLM 综合分析，输出分级报告：
  [普通]   → 建议居家观察 + 护理建议
  [建议就医] → 症状稳定但需专业检查
  [紧急就医] → 生命体征异常/剧痛/大量出血/中毒/呼吸困难
```

**NavigatorAgent 工作流（与 MedicalExpert 协作）**：
```
当 MedicalExpert 输出紧急等级 ≥ [建议就医] 时
       ↓
nearby_hospital_search(用户经纬度坐标)
       ↓
高德地图 POI API 检索 5km 内宠物医院
       ↓
返回：医院名称 + 地址 + 距离 + 联系电话（前5个结果）
```

两个 Agent 的协作体现了"诊断-导航"一体化的完整就医辅助流程。

**使用工具**：`pet_health_knowledge_search`（ChromaDB RAG），`nearby_hospital_search`（高德地图 API）

### 3.6 AuditExpert（首席领养合规性审计专家）

**文件**：`src/agents/audit_expert.py`
**核心职责**：从合规与伦理角度，对领养申请进行独立的第二层语义审查，专注识别"潜在弃养风险"。

**与 AdoptionProfiler 的分工**：

| 维度 | AdoptionProfiler | AuditExpert |
|---|---|---|
| 关注点 | 申请人条件是否匹配宠物需求 | 申请动机是否真实，是否存在欺骗或冲动性申请 |
| 分析方式 | 量化评分 + 多维匹配 | 语义一致性检测 + 逻辑矛盾识别 |
| 输出 | readiness_score + 结构化报告 | 合规意见 + 风险标记 |

**四维度审计**：
1. **空间品种兼容性**：申报住房面积与品种空间需求的实际匹配度
2. **作息陪伴平衡性**：日程安排与宠物依赖程度的冲突检测
3. **经验难度匹配性**：养宠经验与目标品种饲养难度的合理性
4. **语义真实性检测**：识别申请描述中的自相矛盾、过度美化或刻意迎合

### 3.7 PetPersona（宠物拟人化聊天智能体）

**文件**：`src/agents/pet_persona.py` + `src/agents/agents.py`
**核心职责**：以第一人称扮演待领养宠物，与潜在领养人进行情感化对话，提升领养转化率。

**实现特点**：
- **动态角色构建**：每次调用时根据宠物名称、品种、描述动态生成 Agent 的 `role` / `goal` / `backstory`
- **语音合成集成（Edge-TTS）**：每条文字回复同步生成语音，根据品种动态选择音色：
  - 猫咪 / 小型犬 → 晓伊（温柔童声）
  - 大型犬 → 云希（活泼少年音）
  - 其他 → 晓晓（标准女声）
- **回复约束**：强制 50-80 字，适合语音朗读，保持萌化语气
- **情感设计**：宠物会主动表达对温暖家庭的渴望，引发用户情感共鸣

**调用链**：
```
前端点击"和TA聊聊"
     ↓
POST /api/chat (带宠物信息)
     ↓
coordinator.handle_user_input() 判定意图
     ↓
run_pet_chat(user_msg, pet_name, pet_species, pet_desc)
     ↓
PetPersonaAgent 生成文字回复
     ↓
generate_edge_voice(text, pet_species)  ← 并行语音合成
     ↓
返回 { text: "...", audio_base64: "..." }
```

### 3.8 KnowledgeExpert（宠物百科知识专家）

**文件**：`src/web/services/ai_service.py` + `src/agents/tools.py`
**核心职责**：通过 RAG（检索增强生成）技术，将本地知识库与 LLM 结合，提供准确的宠物养护知识问答。

**ChromaDB 知识库构成**：
- `breed_wiki`（品种百科）：50+ 品种的特征、习性、护理要点
- `health_care`（健康护理）：常见疾病、疫苗、日常护理
- `feeding_guide`（喂养指南）：各阶段喂养要点、食材禁忌
- `adoption_knowledge`（领养知识）：领养流程、新宠适应期指导

**RAG 流程**：
```
用户提问
     ↓
pet_health_knowledge_search(keyword)
     ↓
ChromaDB 向量语义检索（Top-5 最相关条目）
     ↓
将检索结果作为上下文注入 LLM Prompt
     ↓
LLM 生成基于知识库的答案（而非"凭空创作"）
     ↓
返回回答 + trace_id（可追溯）
```

## 八、项目亮点总结

### 亮点一：规则引擎 + LLM 混合决策，兼顾可靠性与智能性

领养评估不单纯依赖 LLM"凭感觉"打分，而是先用**确定性规则**过滤明显不合格申请（防止 LLM 被"好听话"欺骗），再用 LLM 进行**语义深度分析**（识别人类难以量化的动机真实性）。最终以 4:6 权重融合，使系统在 LLM 出现幻觉时仍有规则兜底。

### 亮点二：全链路可审计的 AI 决策

所有 AI 智能体的输入、输出、执行时延、降级状态均写入 `agent_trace_logs` 表。管理员可通过后台界面查看任意一次 AI 评估的完整推理过程，解决了 AI 决策"黑盒"问题，为毕业论文中的"可解释性 AI"论点提供了实际工程支撑。

### 亮点三：多层降级机制，保证系统可用性

每个 AI 功能均有降级路径：
- LLM 超时/报错 → 规则引擎兜底
- ChromaDB 检索无结果 → 本地症状字典
- 语义匹配失败 → 关键词模糊匹配

即使 AI 服务全部不可用，系统仍能以基础功能正常运行。

### 亮点四：数据飞轮设计

营养方案 → 用户反馈 → AI 再规划 → 新方案 → 再反馈，形成持续优化的闭环。领养后回访数据 → 信用评分 → 影响未来领养资质评估，形成跨功能模块的数据复用链路。
